# I2C master recommendations for STM32 slave (2026-03-24)

## Контекст

ESP-master работает с STM32 slave в EEPROM-style режиме.

Файл с базовой master-side логикой:
- `docs/l4_i2c_master.c`

Файл с прикладной оркестрацией STM32 packet flow:
- `docs/leo4_stm32_bus.c`

Базовый read-контракт мастера:
- формируется header `[reg_addr, data_len]`;
- затем вызывается `i2c_master_transmit_receive(...)`;
- это создаёт корректную последовательность `write header -> repeated start -> read payload`.

Этот подход признан корректным и совместимым со slave FSM. Менять его не требуется.

---

## Что оказалось полезным на мастере

В обновлённом `docs/leo4_stm32_bus.c` были добавлены паузы между транзакциями.

Подтверждённые места:

1. короткая пауза после сигнала события и перед началом чтения;
2. пауза после чтения `PACKET_TYPE`;
3. пауза перед записью time set (`0x88`);
4. пауза перед записью `I2C_REG_COUNTER_ADDR` (`0x30`);
5. пауза перед записью `I2C_REG_CFG_ADDR` (`0xE0`);
6. пауза в конце цикла обработки.

Именно такая развязка транзакций вместе с исправлениями на slave дала состояние `ошибок нет`.

## Рекомендации по master-side timing

### 1. Сохранять EEPROM-style `transmit_receive()` для read

Рекомендуемая форма read:

- write header: `[reg, len]`
- repeated-start
- read `len` bytes

Это штатно поддерживается slave и не должно заменяться на более сложные последовательности без необходимости.

---

### 2. Не уплотнять две стадии outbox-read

Для slave-initiated пакетов master обычно делает:

1. read `0x00, len=1`
2. read `payload_reg, payload_len`

Рекомендация:
- оставлять небольшой зазор между ними (`BUS_STABILITY_DELAY` или эквивалент 2-10 мс).

Причина:
- slave успевает завершить type-read фазу и безопасно перейти к payload phase;
- снижается риск ложного low-level stuck detection.

---

### 3. Не выполнять write `0x30` и `0xE0` сразу после payload read без паузы

Если после приёма пакета master сразу пишет служебные регистры (`counter`, `cfg`), между операциями должны оставаться паузы.

Рекомендация:
- перед `0x30` оставить `BUS_STABILITY_DELAY`;
- перед `0xE0` оставить `BUS_STABILITY_DELAY`;
- между транзакциями не делать плотного back-to-back burst без задержки.

---

### 4. Для TIME пакета сохранять консервативный режим

Для `PACKET_TIME` мастер выполняет:

1. read `0x00`;
2. read `0x80` (8 байт);
3. при валидном NTP — write `0x88` (7 байт time set);
4. затем служебные write (`0x30`, `0xE0`).

Это допустимо, но только при наличии пауз между стадиями.

---

### 5. Если пакет отсутствует (`PACKET_NULL`) — не делать мгновенный re-poll

Рекомендация:
- если master получил `PACKET_NULL`, не повторять чтение немедленно;
- использовать небольшой backoff (порядка 10-20 мс, либо проектный `BUS_STABILITY_DELAY`).

Это уменьшает ненужную нагрузку на шину и снижает вероятность ложной диагностики на slave.

---

## Рекомендации по error handling на мастере

### 1. Логировать отдельно сбои header-read и payload-read

Полезно различать:
- ошибка чтения `PACKET_TYPE_ADDR`;
- ошибка чтения payload register;
- ошибка последующего write `0x88`, `0x30`, `0xE0`.

Тогда при регрессии проще понять, на каком этапе растёт bus pressure.

---

### 2. Не пытаться «дожимать» шину серией мгновенных retries

Если `i2c_master_transmit_receive()` или `i2c_master_transmit()` вернул ошибку:
- не выполнять tight retry loop;
- лучше использовать ограниченный retry с delay.

Причина:
- tight retry может снова совпасть с recovery-phase slave и ухудшить картину.

---

## Практическое правило для дальнейших изменений

Если потребуется менять master-side логику, безопасный порядок приоритета такой:

1. сначала оставить wire-contract без изменений;
2. затем добавлять/подбирать паузы между транзакциями;
3. только потом трогать более глубокую sequencing logic.

То есть сначала регулировать **timing**, а не **protocol**.

---

## Итог

На текущем этапе подтверждено:

- EEPROM-style read на мастере корректен;
- register contract корректен;
- добавленные паузы между транзакциями полезны и должны быть сохранены;
- сочетание master-side pacing и slave-side conservative recovery привело к состоянию без ошибок.
