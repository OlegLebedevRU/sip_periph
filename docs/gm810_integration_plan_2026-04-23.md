# План интеграции QR-сканера GM810

Дата: 2026-04-23

## 1. Цель

Интегрировать модуль GM810 в `sip_periph` как ещё один источник событий STM32 -> ESP32 со следующим базовым сценарием:

1. GM810 в фоне ожидает код.
2. STM32 принимает данные по UART.
3. STM32 подавляет повторы.
4. STM32 публикует событие в существующую I2C-шину STM32 slave -> ESP32 master.
5. При необходимости STM32 умеет отправлять в GM810 serial-команды для настройки, проверки связи и запуска сканирования.

## 2. Текущие опорные точки в проекте

### 2.1 Уже существующий publish-путь STM32 -> ESP32

В проекте уже есть готовый шаблон для внешних источников данных:

- `Core/Src/service_pn532_task.c` — producer, который получает UID и кладёт пакет в `myQueueToMasterHandle`
- `Core/Src/app_i2c_slave.c`:
  - `StartTaskRxTxI2c1()` — забирает пакет из `myQueueToMasterHandle`
  - `app_i2c_slave_publish()` — публикует payload в `s_ram[]`
  - outbox + `PIN_EVENT_TO_ESP` — сигнализируют ESP32 о готовом пакете
- `docs/i2c_global_contract.md` — нормативный контракт шины STM32 <-> ESP32

Для GM810 логично использовать тот же паттерн: отдельный producer по UART + публикация через существующий I2C outbox.

### 2.2 Выбранная точка подключения

Для GM810 принимаем фиксированное аппаратурное решение:

- использовать `USART6`
- пины STM32: `PA11/PA12`
- ориентир по плате: `docs/chip-pinout-preliminary.md`

Практический смысл такого выбора:

- `USART2` уже занят DWIN HMI
- `USART1` остаётся под текущий RS-485 тракт
- `PA11/PA12` сейчас описаны как USB, но USB для этого сценария опционален
- коннектор на PCB позволяет подключить Reader через этот тракт

Следовательно, в рамках первой интеграции GM810 рассматривается как TTL-UART устройство на `USART6`, а USB-функция на `PA11/PA12` в этом профиле не используется.

Дополнительное уточнение по аппаратурному профилю:

- выбор `USB` vs `USART6` на `PA11/PA12` должен быть явно зафиксирован как build-time опция
- в документации PCB/схемы нужно отметить, что профиль `GM810 over USART6` взаимоисключает USB на этих пинах

## 3. Что важно из manual GM810 по UART-контракту

## 3.1 Физический интерфейс

Из manual:

- питание: `5V`
- сигналы UART:
  - `TX` — выход данных модуля, TTL level
  - `RX` — вход данных в модуль, TTL level
  - `GND`
- если верхний уровень — MCU `3.3V`, TX/RX можно соединять напрямую
- если верхний уровень — PC, нужен преобразователь уровня RS232

Практический вывод для STM32:

- логика должна быть TTL UART, не RS-232
- рабочий тракт проекта фиксируется на `USART6` (`PA11/PA12`)

## 3.2 Параметры UART по умолчанию

Заводские параметры GM810:

- baud rate: `9600`
- data bits: `8`
- stop bits: `1`
- parity: `none`
- flow control: `none`

Поддерживаемые baud rate:

- `1200`, `4800`, `9600`, `14400`, `19200`, `38400`, `57600`, `115200`

Поддерживаемая parity:

- `none`, `odd`, `even`

Рекомендуемая стартовая стратегия:

- первый bring-up делать на `9600 8N1`
- после подтверждения канала при необходимости переводить и GM810, и STM32 на `115200`

## 3.3 Режимы чтения

По manual доступны режимы:

- `Continuous Mode`
- `Induction Mode`
- `Manual Mode`
- `Command Triggered Mode`

Для целевого сценария проекта основным остаётся `Continuous Mode`.

### Вариант A — Continuous Mode

Плюсы:

- максимально соответствует требованию «фоновое ожидание кода»
- минимум управляющего трафика STM32 -> GM810
- simplest runtime model: модуль сам сканирует, STM32 только принимает результат

Минусы:

- выше энергопотребление
- больше риск повторных выдач одного и того же кода без дополнительной дедупликации

### Рекомендация

Для первой интеграции выбрать `Continuous Mode` как основной режим.

`Command Triggered Mode` в проектный профиль не включать.

## 3.4 Защита от повторов на стороне GM810

В manual есть встроенная функция:

- `Same barcode reading delay`
- программируемое окно подавления повтора одного и того же кода

Это полезно использовать, но только как первый уровень фильтрации.

Рекомендуемая роль этой функции:

- на GM810 выставить короткое окно подавления повторов, например `500-1000 ms`
- не полагаться только на GM810
- основную бизнес-дедупликацию оставить в STM32

## 3.5 Формат вывода декодированных данных

GM810 умеет выдавать:

- «чистые данные» (`pure/original data`)
- данные с протоколом
- опциональные `prefix`
- опциональные `suffix`
- опциональный `tail`
- опциональный `AIM ID`
- опциональный `Code ID`

Порядок, описанный в manual:

`[Prefix][AIM ID][CodeID][Data][Suffix][Tail]`

Для проекта принимаем, что QR:

- содержит только печатные ASCII-символы `0x20..0x7E`
- по длине заранее ограничен так, чтобы помещаться в `data`-поле контракта STM32 -> ESP32 без chunking
- в первой версии `data_len <= 12`, а полный publish-пакет для ESP32 фиксирован как `16` байт

### Протокольный вывод

Manual описывает режим:

- output with protocol: `<0x03><length><decoded data>`

Практическое уточнение по фактическому устройству, подтверждённое логическим анализатором на 2026-04-25:

- в используемом проектом профиле GM810 реально выдаёт wire-frame вида `<0x03><0x00><length><decoded data>`
- то есть `0x03` остаётся маркером decoded-data frame, `byte[1]` стабильно равен `0x00`, а длина полезной нагрузки находится в `byte[2]`
- для проектной реализации это наблюдение считается нормативным до появления более точной vendor-спецификации именно для используемого профиля прошивки/настроек модуля
- этот режим работает корректно только при `UTF-8 output format`

Для данного проекта это ограничение приемлемо: в первой версии полезная нагрузка ограничена печатным ASCII, который является безопасным подмножеством `UTF-8`.

### Tail

Поддерживаются варианты tail:

- `CR`
- `CRLF`
- `TAB`
- `None`

### AIM ID

Для QR Code в appendix указан `AIM ID = ]Q1`.

## 3.6 Serial-команды и настройки

Manual описывает serial-протокол команд к модулю.

### Общая рамка

Команды имеют вид:

- header: `7E 00`
- далее идут `type`, `len`, `address`, `data`, `crc`

Ключевые команды:

- `0x07` — read zone bit
- `0x08` — write zone bit
- `0x09` — save to internal flash

Ответ успеха для write/save-команд:

- `02 00 00 01 00 33 31`

### CRC

- CRC = `CRC_CCITT`
- polynomial = `0x1021`
- начальное значение `0`
- manual явно разрешает подставлять `0xAB 0xCD`, если CRC-проверка не нужна

### Временное ограничение команды

Если команда короче ожидаемой или после `7E 00` прошло более `400 ms`, модуль трактует пакет как неизвестную команду и не отвечает.

Практический вывод:

- команду нужно отправлять одним непрерывным кадром
- межбайтовые паузы со стороны STM32 нужно исключить

## 3.7 Полезные команды/возможности из manual

### Heartbeat / calibration

Manual рекомендует периодически слать heartbeat packet:

- запрос: `7E 00 0A 01 00 00 00 30 1A`
- ответ: `03 00 00 01 00 33 31`
- рекомендация manual: раз в `10 s`

Это можно использовать как link-health check, если нужен контроль живости UART-канала.

### Сохранение настроек

После записи zone bit настройки нужно сохранять в flash отдельной командой.

Следствие:

- boot-time конфигурация должна либо всегда завершаться `save`, либо работать как временный runtime override

### Sleep / wake-up

Manual описывает deep sleep и wake-up через serial interrupt.

Для первой интеграции это не нужно и лучше не включать.

## 4. Рекомендуемый рабочий контракт STM32 <-> GM810

## 4.1 Рекомендуемый режим для первой версии

Рекомендуемый профиль GM810 для первой интеграции:

- interface: `TTL UART`
- UART: `9600 8N1` на первом этапе
- read mode: `Continuous Mode`
- same barcode delay: `enabled`
- same barcode delay time: `500-1000 ms`
- output mode: `protocol frame <03><00><len><data>` с payload, ограниченным печатным ASCII `0x20..0x7E`
- tail: `None`
- prefix: `disabled`
- suffix: `disabled`
- AIM ID: `disabled`
- Code ID: `disabled`

## 4.2 Практический формат приёма

### Основной вариант — protocol frame mode

Параметры GM810:

- payload ограничен печатным ASCII `0x20..0x7E`
- output with protocol

STM32 получает frame:

- `0x03`
- `0x00`
- `len`
- `len` байт payload

Плюсы:

- лучший контроль длины кадра
- проще отличать границы сообщений без tail

Минусы:

- в первой версии нужно жёстко отклонять любые байты вне диапазона `0x20..0x7E`
- в первой версии нужно жёстко отклонять payload длиннее `12` байт

### Рекомендация

Так как QR в проекте регламентирован как печатный ASCII, основной рекомендацией является `protocol frame mode`.

`Text line mode` в базовый профиль не закладывать.

## 5. Предложение по приёму данных от GM810 в STM32

## 5.1 Архитектура модуля

Добавить отдельный сервис наподобие `service_pn532_task`, например:

- `Core/Inc/service_gm810_uart.h`
- `Core/Src/service_gm810_uart.c`

Зоны ответственности сервиса:

- владение RX/TX обменом с GM810
- разбор входного UART потока в готовые кадры
- локальная дедупликация
- упаковка результата в `I2cPacketToMaster_t`
- отправка в `myQueueToMasterHandle`

## 5.2 Рекомендуемый способ приёма UART

Предпочтительный вариант:

- `HAL_UARTEx_ReceiveToIdle_DMA(...)` или эквивалентный ring buffer + IDLE detection

Почему:

- GM810 работает как асинхронный producer
- continuous mode хорошо сочетается с DMA + IDLE
- это снизит нагрузку на CPU и упростит приём кадров переменной длины

Если `ReceiveToIdle` по условиям проекта неудобен, fallback-вариант:

- RX interrupt по байту + software ring buffer + parser task

Но для фонового сканера DMA-приём предпочтительнее.

## 5.3 Рекомендуемый pipeline внутри STM32

1. ISR/DMA складывает сырые байты в RX ring.
2. `service_gm810_uart` собирает полный frame.
3. Parser нормализует payload:
   - отделяет protocol header
   - валидирует длину
   - валидирует диапазон символов `0x20..0x7E`
   - валидирует ограничение `data_len <= 12`
4. Нормализованный код проходит дедупликацию.
5. Если код не подавлен дедупликатором — формируется `I2cPacketToMaster_t`.
6. Пакет публикуется через уже существующий `myQueueToMasterHandle -> StartTaskRxTxI2c1() -> app_i2c_slave_publish()`.

## 6. Предложение по дедупликации

Нужна двухуровневая дедупликация.

### Уровень 1 — дедупликация внутри GM810

Использовать встроенный параметр:

- `same barcode reading delay`

Назначение:

- снизить поток одинаковых кадров, пока код неподвижен перед сканером

### Уровень 2 — дедупликация в STM32

В STM32 хранить минимум:

- последний нормализованный payload
- длину payload
- timestamp последней публикации
- опционально: тип/flags кадра

Правило публикации:

- если новый код отличается от последнего — публиковать сразу
- если код совпадает, но suppress-window ещё не истёк — подавлять
- если код совпадает, и suppress-window истёк — публиковать повторно

Рекомендуемое окно STM32-дедупликации:

- стартово `1000-2000 ms`

Это лучше делать независимо от настройки GM810, чтобы поведение системы оставалось управляемым из firmware.

## 6.1 Что считать ключом дедупликации

Минимальный ключ:

- `payload bytes`

Если включён AIM ID или другие добавки, лучше дедуплицировать по уже нормализованным данным после удаления служебных обвязок, иначе одинаковый QR в разных профилях будет считаться разным.

## 6.2 Что делать при длинных кодах

Для первой версии политика фиксируется заранее:

- размер publish-пакета для ESP32: `16` байт
- QR обязан помещаться в `data`-поле этого пакета
- chunked-передача в первой версии не используется
- поля под `chunk_index/chunk_total` оставляются в контракте как резерв на развитие

Если payload не помещается в `data`-поле, это считается нарушением входного регламента и не должно быть штатным сценарием.

Для первой версии поведение при таком случае нужно зафиксировать явно:

- oversize payload отклоняется
- отдельный тип error packet не вводится
- в шину публикуется тот же контрактный пакет `PACKET_QR_GM810`, но с установленным флагом ошибки
- `data_len` при ошибке должен отражать фактически переданную диагностическую нагрузку в рамках того же окна

## 7. Предложение по публикации GM810 в I2C-шину ESP32

## 7.1 Новый packet type

Рекомендуется добавить новый тип пакета, например:

- `PACKET_QR_GM810`

И выделить под него отдельное RAM-окно в свободной зоне, например:

- `0x90`

Причина:

- не пересекать текущие окна `0x00..0x88`, `0xE0`, `0xF0`
- логически отделить QR/serial reader от PN532/Wiegand/HMI

## 7.2 Рекомендуемый формат payload окна

Фиксированное окно `16` байт:

- `byte0` — `data_len`
- `byte1` — `flags`
- `byte2` — `chunk_index`
- `byte3` — `chunk_total`
- `byte4..15` — данные QR (только printable ASCII `0x20..0x7E`)

Где:

- `data_len` — фактическое число байт в `data`
- `chunk_index = 0` для обычного одиночного кадра
- `chunk_total = 1` для обычного одиночного кадра

Где `flags` могут нести:

- bit0 = `from_protocol_mode`
- bit1 = `reserved_chunked`
- bit2 = `error_oversize`
- bit3..7 — reserved

Таким образом:

- общий размер publish-окна = `16` байт
- полезное `data`-поле = `12` байт
- для первой версии допустим только single-frame QR с `data_len <= 12`

### Рекомендация

Для сценария «фоновое ожидание -> дедупликация -> отправка в шину» использовать этот единый формат `16 B` с зарезервированными полями chunking.

## 7.3 Как встроить в текущий I2C-контракт

Потребуются изменения на уровне контракта:

- `Core/Inc/main.h`:
  - новый `PACKET_QR_GM810`
  - новый `I2C_REG_*` для GM810
  - фиксированная длина окна
- `docs/i2c_global_contract.md`:
  - обязательно обновить глобальный контракт: новый `type -> reg -> len`, фиксированное окно `16 B`, printable ASCII `0x20..0x7E`, `data_len <= 12`
- `docs/i2c_gm810_contract.md`:
  - обязательно подготовить отдельный контракт для ESP32
  - документ должен быть читаемым для разработчика и пригодным как prompt/spec для AI-агента, который помогает вносить изменения в проект ESP32

## 8. Варианты отправки команд и настроек из STM32 в GM810

## 8.1 Вариант 1 — без runtime-команд, только ручная преднастройка

Смысл:

- GM810 один раз конфигурируется setup-кодами/сканированием на стенде
- STM32 только принимает готовые данные

Плюсы:

- минимальный объём firmware
- не нужен TX-парсер команд

Минусы:

- состояние модуля зависит от внешней ручной операции
- труднее воспроизводить конфигурацию в производстве/сервисе

## 8.2 Вариант 2 — boot-time инициализация через serial-команды

Смысл:

- при каждом старте STM32 приводит GM810 к целевому профилю serial-командами
- после применения настроек выполняется `save`
- модуль используется только после завершения этой процедуры

Подходит для настройки:

- baud rate
- read mode
- same barcode delay
- tail / suffix / prefix / AIM ID
- protocol mode
- разрешённые типы кодов, если позже появится отдельное бизнес-ограничение

Плюсы:

- детерминированное состояние на каждом старте
- нет зависимости от ручной настройки

Минусы:

- нужен полноценный TX-путь и обработка ответов
- надо аккуратно решить миграцию baud rate, если модуль ещё на `9600`

## 8.3 Вариант 3 — гибридный

Смысл:

- базовые параметры один раз сохраняются в модуле
- STM32 на boot делает короткую проверку/верификацию
- при несовпадении профиля — переприменяет настройки и сохраняет их

Плюсы:

- хороший баланс между простотой и воспроизводимостью
- меньше serial-трафика на каждом старте

### Рекомендация

Для проекта предпочтителен `гибридный вариант`:

- boot-time verify как штатный сценарий
- при несовпадении профиля — автоматическое initial configure + save
- при совпадении профиля — без лишнего конфигурационного трафика

## 8.4 Набор команд, который реально нужен проекту

Минимально полезный набор команд в STM32 API:

1. `gm810_write_zone(addr, data...)`
2. `gm810_read_zone(addr, len)`
3. `gm810_save_config()`
4. `gm810_heartbeat()`
5. `gm810_set_out1(level)` — только если этот вывод потом будет использоваться схемотехнически

## 9. Рекомендуемый поэтапный план внедрения

## Этап 1 — аппаратурная фиксация

- зафиксировать GM810 на `USART6`
- зафиксировать использование `PA11/PA12` под Reader
- зафиксировать, что USB на этом профиле не используется
- зафиксировать `USB/USART6` как build-time опцию для `PA11/PA12` в документации PCB/схемы
- зафиксировать рабочие уровни и питание `5V + TTL`

## Этап 2 — базовый bring-up UART

- поднять приём на `9600 8N1`
- получить сырое чтение кадров `<03><00><len><data>` от GM810 без публикации в I2C
- подтвердить на стенде, что целевой профиль даёт только printable ASCII `0x20..0x7E`

## Этап 3 — минимальный producer в STM32

- добавить сервис `service_gm810_uart`
- принимать кадры
- нормализовать payload
- выводить диагностически в лог/временный буфер

## Этап 4 — локальная дедупликация

- включить same barcode delay в GM810
- добавить STM32 suppress-window
- проверить, что неподвижный QR не спамит очередь

## Этап 5 — публикация в I2C outbox

- добавить новый packet type + RAM window
- отправлять GM810 payload в `myQueueToMasterHandle`
- для oversize payload публиковать тот же `PACKET_QR_GM810` с error flag без отдельного error packet
- обновить `docs/i2c_global_contract.md`
- подготовить `docs/i2c_gm810_contract.md` как отдельный контракт для ESP32; документ должен быть одновременно читаемым для разработчика и пригодным как prompt/spec для AI-агента реализации на стороне ESP32

## 10. Практические решения, которые рекомендуется принять заранее

1. **Источник UART для GM810**
   - зафиксировать `USART6` на `PA11/PA12` как единственный целевой тракт

2. **Формат кадра от GM810**
   - зафиксировать `protocol mode <03><00><len><data>` как обязательный профиль

3. **Максимальная длина QR для I2C-контракта**
   - зафиксировать `16 B` окно и `12 B` data field заранее

4. **Будет ли boot-time auto-config GM810**
   - принять `Hybrid` как штатный режим старта

5. **Нужно ли ограничить decoder только QR-кодами**
   - в первой версии ограничение не включать, так как вероятно нужны и другие штрих-коды

## 11. Рекомендуемое решение для первой реализации

Если нужен самый низкий риск первой интеграции, то целевой профиль должен быть таким:

- GM810 на `USART6`
- STM32 pins = `PA11/PA12`
- GM810 в `Continuous Mode`
- UART `9600 8N1`
- output = `protocol frame <03><00><len><data>` с printable ASCII payload `0x20..0x7E`
- tail = `None`
- prefix/suffix/AIM/CodeID = `off`
- decoder types = `unrestricted`
- same barcode delay = `on`, `500-1000 ms`
- STM32 RX = `DMA + IDLE`
- STM32 dedup window = `1-2 s`
- boot mode = `Hybrid` (verify on boot, configure only on mismatch/uninitialized state)
- новый `PACKET_QR_GM810` публикуется через существующий `myQueueToMasterHandle`
- I2C окно для GM810 — отдельное fixed-length окно `16 B`
- `data`-поле окна = `12 B`, chunk-поля зарезервированы, но в v1 не используются
- oversize payload отклоняется и публикуется тем же `PACKET_QR_GM810` с `flags.error_oversize`
- payload с байтами вне диапазона printable ASCII `0x20..0x7E` отклоняется как невалидный для v1
- при реализации обязательно обновляются `docs/i2c_global_contract.md` и отдельный `docs/i2c_gm810_contract.md` для ESP32/AI-агента

Это решение лучше всего соответствует текущей архитектуре проекта и не ломает существующий transport-подход STM32 -> ESP32.

## 12. Зафиксированные уточнения

1. Отключение USB-профиля при использовании `USART6` на `PA11/PA12` нужно дополнительно фиксировать в документации PCB/схемы как build-time опцию `USB/USART`.
2. В первой версии oversize payload нужно явно отклонять и публиковать тем же контрактным пакетом `PACKET_QR_GM810` с флагом ошибки, без отдельного error packet.
3. Ограничивать decoder только QR-кодами на уровне конфигурации GM810 в первой версии не нужно.
4. В первой версии QR-полезная нагрузка ограничена только печатными ASCII-символами `0x20..0x7E`.
5. В первой версии полный publish-пакет для ESP32 фиксирован как `16 B`, а `data_len <= 12`.
6. При реализации нужно обязательно обновить `docs/i2c_global_contract.md` и подготовить `docs/i2c_gm810_contract.md` как отдельный контракт/spec для ESP32 и AI-агента.

## 13. Итог

GM810 хорошо вписывается в текущую архитектуру проекта, если рассматривать его как ещё один producer событий, аналогичный PN532, но работающий по UART.

Основная рекомендуемая схема такая:

- GM810 непрерывно ждёт код в фоне
- STM32 принимает UART-кадры
- STM32 выполняет локальную дедупликацию
- STM32 публикует событие в существующий I2C outbox для ESP32
- канал STM32 -> GM810 используется для boot-time verify/configuration и heartbeat

Аппаратурная база для первой реализации зафиксирована: `USART6` на `PA11/PA12`, с отказом от USB-функции в этом профиле и с явным оформлением выбора `USB/USART` как build-time опции в документации.

## 14. Конкретные команды для boot mode = Hybrid

Раздел уточняет п. 8.3 + п. 11 (`boot mode = Hybrid`) и фиксирует **точные байтовые команды**, которые STM32 должен слать в GM810 на старте.

Ссылки на manual: `vendor-docs/GM810 Series Barcode reader module User Manual-V1.2.3.pdf`, разделы:

- `1.7 Save and Cancel`
- `2.1 Serial Communication Interface`
- `2.1.2 Serial Port Calibration` (heartbeat)
- `10.1 CRC Algorithm`
- `10.2 Read Zone Bit` (`Type 0x07`)
- `10.3 Write Zone Bit` (`Type 0x08`)
- `10.4 Save Zone Bit To Internal Flash` (`Type 0x09`)
- `10.6 List of zone bit`
- `12 Appendix B: Common serial port instruction`

### 14.1 Принципы

1. Hybrid = `verify on boot, configure only on mismatch/uninitialized state`.
2. На каждом старте STM32 **читает** целевые zone bit и сравнивает побайтно с целевым профилем.
3. Если все целевые байты совпадают — конфигурация считается валидной, никаких write/save не делается.
4. Если хотя бы один байт не совпадает или не получен валидный ответ за окно `400 ms` — выполняется `write` всех целевых zone bit и в конце один `save`.
5. После `write` обязательна команда `save` (10.4), иначе настройки потеряются по питанию.
6. Все команды отправляются одним непрерывным TX-фреймом (без межбайтовых пауз > `400 ms`, см. `10.2/10.3`).
7. CRC-байты приведены готовыми из `Appendix B` где возможно. Для собственных команд допускается `0xAB 0xCD` (manual `10.1`, `10.2`, `10.3`: «when no need for checking CRC, CRC bite can be filled in 0xAB 0xCD»).
8. Boot-обмен идёт на текущей baud rate модуля. Если модуль на не-9600, в первой версии baud-recovery не делается (см. п. 14.6).

### 14.2 Целевой профиль в терминах zone bit

| Zone bit | Адрес    | Целевое значение | Смысл (manual `10.6`)                                                                                                                                              |
| -------- | -------- | ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Mode     | `0x0000` | `0xD6`           | LED on, Mute off, Standard light, Standard brightness, **Continuous Mode** (bit1-0 = `10`). Соответствует `Appendix B` строке «LED on + Mute off + … + Continuous». |
| Baud lo  | `0x002A` | `0x39`           | Baud rate **9600 bps** (`0x0139` little, lo-byte). См. `Appendix B`: `7E 00 08 02 00 2A 39 01 …`.                                                                  |
| Baud hi  | `0x002B` | `0x01`           | Baud rate 9600 bps, hi-byte. Parity = none (bit14-13 = `00`).                                                                                                      |
| Decoder  | `0x002C` | `0x02`           | Full Width + **all bar codes allowed** (decoder unrestricted, см. п.10.5 плана).                                                                                   |
| SBR      | `0x0013` | `0x85`           | Same Barcode Reading delay **on** (bit7=1) + delay = `0x05` × `100 ms` = **500 ms**. Совпадает с п.4.1 плана (500–1000 ms, выбираем нижнюю границу).               |
| Output   | `0x0060` | `0xE0`           | bit7=1 **with protocol** + bit6-5=`11` tail=None + bit4=0 RF off + bit3=0 prefix off + bit2=0 CodeID off + bit1=0 suffix off + bit0=0 tail off.                    |
| AIM ID   | `0x00D0` | `0x00`           | **Forbid AIM ID** (см. `Appendix B`: «Forbid AIM ID = 7E 00 08 01 00 D0 00 AB CD»).                                                                                |

Эти 7 zone bit покрывают все требования п. 11 плана. Остальные параметры остаются заводскими.

### 14.3 Read zone bit — verify-команды

Формат (`10.2`): `7E 00 07 01 <addrH> <addrL> <count> <CRC1> <CRC2>`. CRC заменяется на `AB CD` где допустимо.

Ожидаемый ответ: `02 00 00 <len> <data...> <CRC1> <CRC2>` (данные не сравниваем, кроме байтов `data...`).

| Цель                         | TX-команда (hex)                  | Ожидаемые `data` в ответе |
| ---------------------------- | --------------------------------- | ------------------------- |
| Read mode (`0x0000`)         | `7E 00 07 01 00 00 01 AB CD`      | `D6`                      |
| Read SBR (`0x0013`)          | `7E 00 07 01 00 13 01 AB CD`      | `85`                      |
| Read baud `0x002A` + `0x002B`| `7E 00 07 01 00 2A 02 AB CD`      | `39 01`                   |
| Read decoder (`0x002C`)      | `7E 00 07 01 00 2C 01 AB CD`      | `02`                      |
| Read output (`0x0060`)       | `7E 00 07 01 00 60 01 AB CD`      | `E0`                      |
| Read AIM ID (`0x00D0`)       | `7E 00 07 01 00 D0 01 AB CD`      | `00`                      |

Готовая верификационная команда «Find baud rate» из `Appendix B` (`7E 00 07 01 00 2A 02 D8 0F`) даёт тот же ответ, что и read `0x002A` длиной 2; STM32 может использовать её как fallback, но в общем случае достаточно read `0x002A` len `2`.

Алгоритм verify:

1. Для каждой строки таблицы 14.3 — отправить TX-команду одним кадром.
2. Ждать `RX` ответа `02 00 …` максимум `200 ms` (manual оговаривает только тайм-аут на TX-сторону команд `400 ms`; `200 ms` — практический READ-таймаут с запасом).
3. Если ответ пришёл и `data...` совпадает с ожидаемым — этот zone bit OK.
4. Если ответ не пришёл или не совпадает хотя бы один байт — пометить `mismatch_or_uninitialized = true`.
5. Если `mismatch_or_uninitialized == false` после всех чтений — boot-конфигурацию **не трогать**, перейти к разрешению парсера.

### 14.4 Write zone bit — configure-команды (запускаются только при mismatch)

Формат (`10.3`): `7E 00 08 <len> <addrH> <addrL> <data...> AB CD`, где `<len>` = число байт `data...`.

Порядок отправки выбран от низкого адреса к высокому (требование `10.3`: «must follow the order of address from low to high»). Каждая команда — отдельный непрерывный кадр; между кадрами допускается пауза, но в одном кадре пауз быть не должно.

| Шаг | Назначение                              | TX-команда (hex)                            |
| --- | --------------------------------------- | ------------------------------------------- |
| W1  | Mode = `0xD6` (Continuous, LED on, …)   | `7E 00 08 01 00 00 D6 AB CD`                |
| W2  | Same Barcode Delay on, 500 ms           | `7E 00 08 01 00 13 85 AB CD`                |
| W3  | Baud = 9600 (одной 2-байтовой записью)  | `7E 00 08 02 00 2A 39 01 AB CD`             |
| W4  | Decoder = full + all codes allowed      | `7E 00 08 01 00 2C 02 AB CD`                |
| W5  | Output = with-protocol, tail/prefix/etc | `7E 00 08 01 00 60 E0 AB CD`                |
| W6  | AIM ID = forbid                         | `7E 00 08 01 00 D0 00 AB CD`                |

Ожидаемый ответ на каждый успешный write (`10.3`): `02 00 00 01 00 33 31`.

После всех успешных write — ровно один **save** (`10.4`):

| Шаг | Назначение                          | TX-команда (hex)                |
| --- | ----------------------------------- | ------------------------------- |
| S1  | Save zone bit list to internal Flash | `7E 00 09 01 00 00 00 DE C8`    |

Здесь CRC `DE C8` берётся из `Appendix B` как готовое значение (это валидный `CRC_CCITT` для `09 01 00 00 00`; `AB CD` тоже допустим, но `DE C8` оставляем как «предпочитаемый», совпадающий с manual).

Ожидаемый ответ на save: `02 00 00 01 00 33 31`.

### 14.5 Опционально: heartbeat как health-check

Из manual `2.1.2`:

- TX: `7E 00 0A 01 00 00 00 30 1A`
- RX: `03 00 00 01 00 33 31`
- рекомендуемый период: `10 s`, разрешено `3` пропуска подряд до признания канала битым.

В первой версии heartbeat **не обязателен** для boot-mode, но удобен как пост-boot link-monitor. Если он включается, его TX-фреймы должны идти с тем же требованием «один непрерывный кадр».

### 14.6 Граничные случаи и поведение

1. **Модуль не отвечает на `read 0x002A` len 2**: считаем, что модуль либо на другом baud rate, либо отсутствует. В первой версии — пометить boot как `failed`, оставить парсер выключенным, инкрементировать диагностический счётчик. Авто-обход baud rate (1200/4800/14400/19200/38400/57600/115200) — задача отдельного этапа, **в v1 не делать**.
2. **Read вернул валидный кадр, но данные не совпали**: запустить полный configure (W1…W6) + один save (S1). Повторно перечитать (verify) и только при успешном повторном verify признать boot OK.
3. **Write подтверждён, save не подтверждён**: пометить boot как `failed_save`, парсер **не запускать** (иначе следующий power-cycle вернёт устройство к старой конфигурации без нашего ведома).
4. **Любая команда не уложилась в `400 ms` из-за GM810_FRAME_TIMEOUT**: повторить TX до `2` раз; при стойком фейле — boot `failed`.
5. **Поток данных из `Continuous Mode` начинает приходить во время verify/configure**: TX-фреймы команд имеют приоритет; RX-парсер должен быть **выключен на всё время boot-procedure**, иначе фрагменты сканированного кода будут интерпретироваться как «битый ответ». Включать парсер только после успешного завершения boot.
6. **Power-on settle**: текущий код уже выдерживает `GM810_BOOT_READY_DELAY_MS = 300 ms` перед стартом RX и flush'ит ранний шум; verify-фазу запускать **после** этого окна.

### 14.7 Сводная диаграмма последовательности (Hybrid)

```
service_gm810_uart_init()
        │
        ▼
service_gm810_uart_start()        // существующий entry point
        │
        ▼
delay 300 ms  +  flush RX         // уже реализовано
        │
        ▼
boot_verify():                    // НОВОЕ
    for each row in §14.3:
        TX read zone bit
        wait RX (≤200 ms)
        compare bytes
    if all match → goto ARM_PARSER
        │
        ▼
boot_configure():                 // НОВОЕ
    TX W1 … W6 (§14.4)
    TX S1 save (§14.4)
    wait & validate each ACK = 02 00 00 01 00 33 31
        │
        ▼
boot_verify() again               // perevereka
    if mismatch → boot = failed, parser off
        │
        ▼
ARM_PARSER:
    HAL_UART_Receive_IT(...)      // существующий путь приёма
```

Раздел §14 этого документа фиксирует точные команды, которые должна выполнить будущая boot-procedure.

Дополнительные расхождения, которые **не блокируют** boot-mode, но стоит зафиксировать отдельно:

1. RX-приём через `HAL_UART_Receive_IT` len=1, а не `HAL_UARTEx_ReceiveToIdle_DMA`. Для boot-фазы это не критично (TX-обмен короткий и синхронный), но для долгосрочного приёма continuous-потока DMA+IDLE остаётся предпочтительнее (см. §5.2).
2. Политика по non-printable байтам в payload в коде «мягче» текста §11. Решение — либо ослабить формулировку §11 (текущий код корректен и публикует флаг ошибки), либо ужесточить код (полное отклонение пакета без публикации). Этот выбор — вне скоупа boot-mode и должен быть сделан отдельно.

**Title**: Реализовать GM810 boot-mode = Hybrid (verify-on-boot, configure-on-mismatch, save) в `sip_periph`.

**Context (read-only)**:

- Репозиторий: `OlegLebedevRU/sip_periph` (STM32F411, FreeRTOS, HAL).
- Целевая постановка и точные байтовые команды: `docs/gm810_integration_plan_2026-04-23.md`, разделы **§4.1**, **§8.3**, **§11**, **§14** (целевой профиль и команды), **§15** (audit текущего кода).
- Vendor reference: `vendor-docs/GM810 Series Barcode reader module User Manual-V1.2.3.pdf`, главы **10.1–10.6**, **2.1.2**, **Appendix B**.
- Текущий сервис: `Core/Inc/service_gm810_uart.h`, `Core/Src/service_gm810_uart.c`.
- Контракт окна: `Core/Inc/main.h` (`PACKET_QR_GM810`, `I2C_PACKET_QR_GM810_LEN`, `GM810_QR_DATA_MAX_LEN`, `GM810_QR_FLAG_*`).
- UART init: `Core/Src/main.c` функция `MX_USART6_UART_Init` (9600 8N1, no flow). **Менять не требуется.**
- Producer-паттерн (для аналогии с PN532): `Core/Src/service_pn532_task.c`.

**Goal**:

Добавить TX-канал в `service_gm810_uart` и встроить в существующий `service_gm810_uart_start()` процедуру Hybrid-boot, чтобы исключить зависимость от ручной преднастройки модуля. Парсер RX и публикация в `myQueueToMasterHandle` остаются без изменений по контракту.

**Functional requirements**:

1. Реализовать минимальный TX API в `service_gm810_uart` (внутренний static, экспорт по необходимости только для unit-тестов/диагностики):
   - `gm810_tx_frame(const uint8_t *bytes, size_t len, uint32_t timeout_ms)` — синхронная отправка одним непрерывным кадром через `HAL_UART_Transmit` (без межбайтовых пауз). Тайм-аут не больше `100 ms` на типовой 9-байтный кадр.
   - `gm810_read_zone(uint16_t addr, uint8_t count, uint8_t *out_data, uint32_t resp_timeout_ms)` — формирует команду `7E 00 07 01 <addrH> <addrL> <count> AB CD`, шлёт, ждёт ответ `02 00 00 <len> <data...> <crc1> <crc2>`. Возвращает 0 при успехе и валидной длине. CRC ответа не верифицировать в v1 (manual это допускает).
   - `gm810_write_zone(uint16_t addr, const uint8_t *data, uint8_t len)` — формирует `7E 00 08 <len> <addrH> <addrL> <data...> AB CD`, ждёт ACK `02 00 00 01 00 33 31`.
   - `gm810_save_config(void)` — шлёт `7E 00 09 01 00 00 00 DE C8`, ждёт ACK `02 00 00 01 00 33 31`.

2. Реализовать boot-procedure внутри `service_gm810_uart_start()` **после** существующего `delay 300 ms + flush + parser_reset` и **до** первого `HAL_UART_Receive_IT`:
   - `boot_verify()` — последовательно читает 7 zone bit из таблицы §14.3 и сверяет с целевыми значениями.
   - Если всё совпало — boot OK, сразу армировать парсер (как сейчас).
   - Если хотя бы одно несоответствие или RX-таймаут — `boot_configure()` шлёт W1…W6 из §14.4, затем S1 (save), затем повторно `boot_verify()`. Только при успешном повторном verify — boot OK.
   - При двух подряд провалах verify — boot `failed`, парсер **не армировать**, инкрементировать поле диагностики `boot_failures`.

3. Запретить параллельную работу RX-парсера и TX-команд:
   - На время `boot_verify`/`boot_configure` использовать `HAL_UART_AbortReceive` (он уже вызывается в `service_gm810_uart_start`); приём байта парсером арммировать только после завершения boot.
   - Для RX ответа на TX-команду использовать **локальный** `HAL_UART_Receive` с тайм-аутом `≤ 200 ms`, не трогая глобальный `s_rx_byte` парсера.

4. Расширить `service_gm810_uart_diag_t` в `Core/Inc/service_gm810_uart.h`:
   - `uint32_t boot_verify_calls;`
   - `uint32_t boot_configure_calls;`
   - `uint32_t boot_save_calls;`
   - `uint32_t boot_failures;`
   - `uint32_t boot_last_mismatch_addr;`  // адрес первого несовпавшего zone bit, для диагностики
   - `uint8_t  boot_state;`              // 0=idle, 1=verify, 2=configure, 3=ok, 4=failed

5. Не вводить heartbeat в этой задаче. Заложить только TODO-комментарий с ссылкой на §14.5.

**Non-functional / contract**:

- НЕ менять контракт `PACKET_QR_GM810`, `I2C_PACKET_QR_GM810_LEN`, `GM810_QR_DATA_MAX_LEN`, `GM810_QR_FLAG_*`.
- НЕ менять `MX_USART6_UART_Init` (baud остаётся 9600, см. §14.6 п.1 — авто-обход baud не делать).
- НЕ менять `docs/i2c_global_contract.md` и `docs/i2c_gm810_contract.md` — boot-mode не задевает I2C-контракт.
- НЕ переходить с `HAL_UART_Receive_IT` на DMA+IDLE в этой задаче (это отдельный refactor по §5.2).
- Все hex-команды копировать **строго** из таблиц §14.3 и §14.4 настоящего документа. Не пересчитывать CRC для готовых команд из Appendix B (W3 baud, S1 save, AIM ID forbid и т.д.).
- Между байтами одной команды не должно быть пауз > 400 ms (manual `10.2`/`10.3`). Это обеспечивается `HAL_UART_Transmit` одним вызовом — никакой ручной побайтной отправки.

**Files expected to change**:

- `Core/Inc/service_gm810_uart.h` — расширение `service_gm810_uart_diag_t`.
- `Core/Src/service_gm810_uart.c` — TX API, boot-procedure, интеграция в `service_gm810_uart_start()`.

**Files NOT to change**:

- `Core/Inc/main.h` (контрактные макросы `PACKET_QR_GM810`/`I2C_PACKET_QR_GM810_LEN`/`GM810_QR_DATA_MAX_LEN`/`GM810_QR_FLAG_*` и pin-defines — без изменений).
- `Core/Src/main.c` (`MX_USART6_UART_Init`, `HAL_UART_*Callback` диспетчеризация — без изменений).
- `docs/i2c_global_contract.md`, `docs/i2c_gm810_contract.md`.

**Acceptance criteria**:

1. После cold boot модуля с заведомо «чужими» настройками (например, suffix включён, tail=CR) STM32 за один цикл `service_gm810_uart_start()` приводит модуль к целевому профилю §14.2 и записывает в flash; следующий cold boot завершается без write/save (только verify).
2. После cold boot модуля с уже целевыми настройками `boot_configure_calls` и `boot_save_calls` остаются `0`; `boot_state == 3 (ok)`.
3. При физически отключённом GM810 `service_gm810_uart_start()` завершается с `boot_state == 4 (failed)` и **не** армирует RX-парсер; основная FreeRTOS-задача системы при этом не блокируется навсегда (тайм-ауты соблюдены).
4. Существующий путь публикации `PACKET_QR_GM810` через `myQueueToMasterHandle` работает без изменений на штатно сконфигурированном модуле (regression check: имитировать valid `<03><len><payload>` — пакет должен попасть в очередь как и раньше).
5. Сборка проекта проходит существующими средствами репо без новых warning'ов в файлах `service_gm810_uart.*`.

**Out of scope (явно не делать в этой задаче)**:

- Авто-определение/изменение baud rate (см. §14.6 п.1).
- Heartbeat-таск (§14.5).
- Переход RX на DMA+IDLE (§5.2).
- Изменение политики по non-printable payload (см. §15, последний пункт замечаний).
- Любые изменения в i2c-контракте и в публикации.
