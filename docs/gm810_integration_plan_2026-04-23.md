# План интеграции QR-сканера GM810

Дата: 2026-04-23

## 1. Цель

Интегрировать модуль GM810 в `sip_periph` как ещё один источник событий STM32 -> ESP32 со следующим базовым сценарием:

1. GM810 в фоне ожидает код.
2. STM32 принимает данные по UART.
3. STM32 подавляет повторы.
4. STM32 публикует событие в существующую I2C-шину STM32 slave -> ESP32 master.
5. При необходимости STM32 умеет отправлять в GM810 serial-команды для настройки, проверки связи и запуска сканирования.

## 2. Текущие опорные точки в проекте

### 2.1 Уже существующий publish-путь STM32 -> ESP32

В проекте уже есть готовый шаблон для внешних источников данных:

- `Core/Src/service_pn532_task.c` — producer, который получает UID и кладёт пакет в `myQueueToMasterHandle`
- `Core/Src/app_i2c_slave.c`:
  - `StartTaskRxTxI2c1()` — забирает пакет из `myQueueToMasterHandle`
  - `app_i2c_slave_publish()` — публикует payload в `s_ram[]`
  - outbox + `PIN_EVENT_TO_ESP` — сигнализируют ESP32 о готовом пакете
- `docs/i2c_global_contract.md` — нормативный контракт шины STM32 <-> ESP32

Для GM810 логично использовать тот же паттерн: отдельный producer по UART + публикация через существующий I2C outbox.

### 2.2 Выбранная точка подключения

Для GM810 принимаем фиксированное аппаратурное решение:

- использовать `USART6`
- пины STM32: `PA11/PA12`
- ориентир по плате: `docs/chip-pinout-preliminary.md`

Практический смысл такого выбора:

- `USART2` уже занят DWIN HMI
- `USART1` остаётся под текущий RS-485 тракт
- `PA11/PA12` сейчас описаны как USB, но USB для этого сценария опционален
- коннектор на PCB позволяет подключить Reader через этот тракт

Следовательно, в рамках первой интеграции GM810 рассматривается как TTL-UART устройство на `USART6`, а USB-функция на `PA11/PA12` в этом профиле не используется.

## 3. Что важно из manual GM810 по UART-контракту

## 3.1 Физический интерфейс

Из manual:

- питание: `5V`
- сигналы UART:
  - `TX` — выход данных модуля, TTL level
  - `RX` — вход данных в модуль, TTL level
  - `GND`
- если верхний уровень — MCU `3.3V`, TX/RX можно соединять напрямую
- если верхний уровень — PC, нужен преобразователь уровня RS232

Практический вывод для STM32:

- логика должна быть TTL UART, не RS-232
- рабочий тракт проекта фиксируется на `USART6` (`PA11/PA12`)

## 3.2 Параметры UART по умолчанию

Заводские параметры GM810:

- baud rate: `9600`
- data bits: `8`
- stop bits: `1`
- parity: `none`
- flow control: `none`

Поддерживаемые baud rate:

- `1200`, `4800`, `9600`, `14400`, `19200`, `38400`, `57600`, `115200`

Поддерживаемая parity:

- `none`, `odd`, `even`

Рекомендуемая стартовая стратегия:

- первый bring-up делать на `9600 8N1`
- после подтверждения канала при необходимости переводить и GM810, и STM32 на `115200`

## 3.3 Режимы чтения

По manual доступны режимы:

- `Continuous Mode`
- `Induction Mode`
- `Manual Mode`
- `Command Triggered Mode`

Для целевого сценария проекта основным остаётся `Continuous Mode`.

### Вариант A — Continuous Mode

Плюсы:

- максимально соответствует требованию «фоновое ожидание кода»
- минимум управляющего трафика STM32 -> GM810
- simplest runtime model: модуль сам сканирует, STM32 только принимает результат

Минусы:

- выше энергопотребление
- больше риск повторных выдач одного и того же кода без дополнительной дедупликации

### Рекомендация

Для первой интеграции выбрать `Continuous Mode` как основной режим.

`Command Triggered Mode` в проектный профиль не включать.

## 3.4 Защита от повторов на стороне GM810

В manual есть встроенная функция:

- `Same barcode reading delay`
- программируемое окно подавления повтора одного и того же кода

Это полезно использовать, но только как первый уровень фильтрации.

Рекомендуемая роль этой функции:

- на GM810 выставить короткое окно подавления повторов, например `500-1000 ms`
- не полагаться только на GM810
- основную бизнес-дедупликацию оставить в STM32

## 3.5 Формат вывода декодированных данных

GM810 умеет выдавать:

- «чистые данные» (`pure/original data`)
- данные с протоколом
- опциональные `prefix`
- опциональные `suffix`
- опциональный `tail`
- опциональный `AIM ID`
- опциональный `Code ID`

Порядок, описанный в manual:

`[Prefix][AIM ID][CodeID][Data][Suffix][Tail]`

Для проекта принимаем, что QR:

- всегда приходит как корректный `UTF-8`
- по длине заранее ограничен так, чтобы помещаться в `data`-поле контракта STM32 -> ESP32 без chunking

### Протокольный вывод

Manual описывает режим:

- output with protocol: `<0x03><length><decoded data>`
- этот режим работает корректно только при `UTF-8 output format`

Для данного проекта это ограничение приемлемо и соответствует целевому контракту.

### Tail

Поддерживаются варианты tail:

- `CR`
- `CRLF`
- `TAB`
- `None`

### AIM ID

Для QR Code в appendix указан `AIM ID = ]Q1`.

## 3.6 Serial-команды и настройки

Manual описывает serial-протокол команд к модулю.

### Общая рамка

Команды имеют вид:

- header: `7E 00`
- далее идут `type`, `len`, `address`, `data`, `crc`

Ключевые команды:

- `0x07` — read zone bit
- `0x08` — write zone bit
- `0x09` — save to internal flash

Ответ успеха для write/save-команд:

- `02 00 00 01 00 33 31`

### CRC

- CRC = `CRC_CCITT`
- polynomial = `0x1021`
- начальное значение `0`
- manual явно разрешает подставлять `0xAB 0xCD`, если CRC-проверка не нужна

### Временное ограничение команды

Если команда короче ожидаемой или после `7E 00` прошло более `400 ms`, модуль трактует пакет как неизвестную команду и не отвечает.

Практический вывод:

- команду нужно отправлять одним непрерывным кадром
- межбайтовые паузы со стороны STM32 нужно исключить

## 3.7 Полезные команды/возможности из manual

### Heartbeat / calibration

Manual рекомендует периодически слать heartbeat packet:

- запрос: `7E 00 0A 01 00 00 00 30 1A`
- ответ: `03 00 00 01 00 33 31`
- рекомендация manual: раз в `10 s`

Это можно использовать как link-health check, если нужен контроль живости UART-канала.

### Сохранение настроек

После записи zone bit настройки нужно сохранять в flash отдельной командой.

Следствие:

- boot-time конфигурация должна либо всегда завершаться `save`, либо работать как временный runtime override

### Sleep / wake-up

Manual описывает deep sleep и wake-up через serial interrupt.

Для первой интеграции это не нужно и лучше не включать.

## 4. Рекомендуемый рабочий контракт STM32 <-> GM810

## 4.1 Рекомендуемый режим для первой версии

Рекомендуемый профиль GM810 для первой интеграции:

- interface: `TTL UART`
- UART: `9600 8N1` на первом этапе
- read mode: `Continuous Mode`
- same barcode delay: `enabled`
- same barcode delay time: `500-1000 ms`
- output mode: `UTF-8 + protocol frame <03><len><data>`
- tail: `None`
- prefix: `disabled`
- suffix: `disabled`
- AIM ID: `disabled`
- Code ID: `disabled`

## 4.2 Практический формат приёма

### Основной вариант — protocol frame mode

Параметры GM810:

- UTF-8 output
- output with protocol

STM32 получает frame:

- `0x03`
- `len`
- `len` байт payload

Плюсы:

- лучший контроль длины кадра
- проще отличать границы сообщений без tail

Минусы:

- режим привязан к UTF-8 output format
- если ожидаются нетекстовые QR payload, нужно отдельно подтвердить, что этот режим не ломает данные

### Рекомендация

Так как QR в проекте регламентирован как текстовый `UTF-8`, основной рекомендацией является `protocol frame mode`.

`Text line mode` в базовый профиль не закладывать.

## 5. Предложение по приёму данных от GM810 в STM32

## 5.1 Архитектура модуля

Добавить отдельный сервис наподобие `service_pn532_task`, например:

- `Core/Inc/service_gm810_uart.h`
- `Core/Src/service_gm810_uart.c`

Зоны ответственности сервиса:

- владение RX/TX обменом с GM810
- разбор входного UART потока в готовые кадры
- локальная дедупликация
- упаковка результата в `I2cPacketToMaster_t`
- отправка в `myQueueToMasterHandle`

## 5.2 Рекомендуемый способ приёма UART

Предпочтительный вариант:

- `HAL_UARTEx_ReceiveToIdle_DMA(...)` или эквивалентный ring buffer + IDLE detection

Почему:

- GM810 работает как асинхронный producer
- continuous mode хорошо сочетается с DMA + IDLE
- это снизит нагрузку на CPU и упростит приём кадров переменной длины

Если `ReceiveToIdle` по условиям проекта неудобен, fallback-вариант:

- RX interrupt по байту + software ring buffer + parser task

Но для фонового сканера DMA-приём предпочтительнее.

## 5.3 Рекомендуемый pipeline внутри STM32

1. ISR/DMA складывает сырые байты в RX ring.
2. `service_gm810_uart` собирает полный frame.
3. Parser нормализует payload:
   - отделяет protocol header
   - валидирует длину
   - валидирует корректность `UTF-8`
4. Нормализованный код проходит дедупликацию.
5. Если код не подавлен дедупликатором — формируется `I2cPacketToMaster_t`.
6. Пакет публикуется через уже существующий `myQueueToMasterHandle -> StartTaskRxTxI2c1() -> app_i2c_slave_publish()`.

## 6. Предложение по дедупликации

Нужна двухуровневая дедупликация.

### Уровень 1 — дедупликация внутри GM810

Использовать встроенный параметр:

- `same barcode reading delay`

Назначение:

- снизить поток одинаковых кадров, пока код неподвижен перед сканером

### Уровень 2 — дедупликация в STM32

В STM32 хранить минимум:

- последний нормализованный payload
- длину payload
- timestamp последней публикации
- опционально: тип/flags кадра

Правило публикации:

- если новый код отличается от последнего — публиковать сразу
- если код совпадает, но suppress-window ещё не истёк — подавлять
- если код совпадает, и suppress-window истёк — публиковать повторно

Рекомендуемое окно STM32-дедупликации:

- стартово `1000-2000 ms`

Это лучше делать независимо от настройки GM810, чтобы поведение системы оставалось управляемым из firmware.

## 6.1 Что считать ключом дедупликации

Минимальный ключ:

- `payload bytes`

Если включён AIM ID или другие добавки, лучше дедуплицировать по уже нормализованным данным после удаления служебных обвязок, иначе одинаковый QR в разных профилях будет считаться разным.

## 6.2 Что делать при длинных кодах

Для первой версии политика фиксируется заранее:

- размер publish-пакета для ESP32: `32` байта
- QR обязан помещаться в `data`-поле этого пакета
- chunked-передача в первой версии не используется
- поля под `chunk_index/chunk_total` оставляются в контракте как резерв на развитие

Если payload не помещается в `data`-поле, это считается нарушением входного регламента и не должно быть штатным сценарием.

## 7. Предложение по публикации GM810 в I2C-шину ESP32

## 7.1 Новый packet type

Рекомендуется добавить новый тип пакета, например:

- `PACKET_QR_GM810`

И выделить под него отдельное RAM-окно в свободной зоне, например:

- `0x90`

Причина:

- не пересекать текущие окна `0x00..0x88`, `0xE0`, `0xF0`
- логически отделить QR/serial reader от PN532/Wiegand/HMI

## 7.2 Рекомендуемый формат payload окна

Фиксированное окно `32` байта:

- `byte0` — `data_len`
- `byte1` — `flags`
- `byte2` — `chunk_index`
- `byte3` — `chunk_total`
- `byte4..31` — данные QR (`UTF-8`)

Где:

- `data_len` — фактическое число байт в `data`
- `chunk_index = 0` для обычного одиночного кадра
- `chunk_total = 1` для обычного одиночного кадра

Где `flags` могут нести:

- bit0 = `from_protocol_mode`
- bit1 = `reserved_chunked`
- bit2..7 — reserved

Таким образом:

- общий размер publish-окна = `32` байта
- полезное `data`-поле = `28` байт
- для первой версии допустим только single-frame QR с `data_len <= 28`

### Рекомендация

Для сценария «фоновое ожидание -> дедупликация -> отправка в шину» использовать этот единый формат `32 B` с зарезервированными полями chunking.

## 7.3 Как встроить в текущий I2C-контракт

Потребуются изменения на уровне контракта:

- `Core/Inc/main.h`:
  - новый `PACKET_QR_GM810`
  - новый `I2C_REG_*` для GM810
  - фиксированная длина окна
- `docs/i2c_global_contract.md`:
  - новый `type -> reg -> len`
- возможно отдельный scenario doc по аналогии с `docs/i2c_uid532_contract.md`

## 8. Варианты отправки команд и настроек из STM32 в GM810

## 8.1 Вариант 1 — без runtime-команд, только ручная преднастройка

Смысл:

- GM810 один раз конфигурируется setup-кодами/сканированием на стенде
- STM32 только принимает готовые данные

Плюсы:

- минимальный объём firmware
- не нужен TX-парсер команд

Минусы:

- состояние модуля зависит от внешней ручной операции
- труднее воспроизводить конфигурацию в производстве/сервисе

## 8.2 Вариант 2 — boot-time инициализация через serial-команды

Смысл:

- при каждом старте STM32 приводит GM810 к целевому профилю serial-командами
- после применения настроек выполняется `save`
- модуль используется только после завершения этой процедуры

Подходит для настройки:

- baud rate
- read mode
- same barcode delay
- tail / suffix / prefix / AIM ID
- protocol mode
- разрешённые типы кодов (например, оставить только QR, если это допустимо бизнесом)

Плюсы:

- детерминированное состояние на каждом старте
- нет зависимости от ручной настройки

Минусы:

- нужен полноценный TX-путь и обработка ответов
- надо аккуратно решить миграцию baud rate, если модуль ещё на `9600`

## 8.3 Вариант 3 — гибридный

Смысл:

- базовые параметры один раз сохраняются в модуле
- STM32 на boot делает короткую проверку/верификацию
- при несовпадении профиля — переприменяет настройки и сохраняет их

Плюсы:

- хороший баланс между простотой и воспроизводимостью
- меньше serial-трафика на каждом старте

### Рекомендация

Для проекта предпочтителен `гибридный вариант`:

- boot-time verify как штатный сценарий
- при несовпадении профиля — автоматическое initial configure + save
- при совпадении профиля — без лишнего конфигурационного трафика

## 8.4 Набор команд, который реально нужен проекту

Минимально полезный набор команд в STM32 API:

1. `gm810_write_zone(addr, data...)`
2. `gm810_read_zone(addr, len)`
3. `gm810_save_config()`
4. `gm810_heartbeat()`
5. `gm810_set_out1(level)` — только если этот вывод потом будет использоваться схемотехнически

## 9. Рекомендуемый поэтапный план внедрения

## Этап 1 — аппаратурная фиксация

- зафиксировать GM810 на `USART6`
- зафиксировать использование `PA11/PA12` под Reader
- зафиксировать, что USB на этом профиле не используется
- зафиксировать рабочие уровни и питание `5V + TTL`

## Этап 2 — базовый bring-up UART

- поднять приём на `9600 8N1`
- получить сырое чтение кадров `<03><len><data>` от GM810 без публикации в I2C
- подтвердить корректность `UTF-8` payload на стенде

## Этап 3 — минимальный producer в STM32

- добавить сервис `service_gm810_uart`
- принимать кадры
- нормализовать payload
- выводить диагностически в лог/временный буфер

## Этап 4 — локальная дедупликация

- включить same barcode delay в GM810
- добавить STM32 suppress-window
- проверить, что неподвижный QR не спамит очередь

## Этап 5 — публикация в I2C outbox

- добавить новый packet type + RAM window
- отправлять GM810 payload в `myQueueToMasterHandle`
- обновить I2C contract docs

## Этап 6 — обратный канал команд

- добавить TX API для serial-команд
- реализовать read/write/save/heartbeat
- решить, какие параметры конфигурируются на boot

## Этап 7 — интеграционные проверки с ESP32

- ESP32 получает новый `type`
- читает выделенное окно
- корректно обрабатывает дедуплицированный поток
- проверяется поведение при длинном QR, пустом ответе, обрыве UART, повторном чтении

## 10. Практические решения, которые рекомендуется принять заранее

1. **Источник UART для GM810**
   - зафиксировать `USART6` на `PA11/PA12` как единственный целевой тракт

2. **Формат кадра от GM810**
   - зафиксировать `protocol mode <03><len><data>` как обязательный профиль

3. **Максимальная длина QR для I2C-контракта**
   - зафиксировать `32 B` окно и `28 B` data field заранее

4. **Будет ли boot-time auto-config GM810**
   - принять `Hybrid` как штатный режим старта

5. **Нужно ли ограничить decoder только QR-кодами**
   - если сценарий только QR, это уменьшит шум и ложные распознавания

## 11. Рекомендуемое решение для первой реализации

Если нужен самый низкий риск первой интеграции, то целевой профиль должен быть таким:

- GM810 на `USART6`
- STM32 pins = `PA11/PA12`
- GM810 в `Continuous Mode`
- UART `9600 8N1`
- output = `UTF-8 + protocol frame <03><len><data>`
- tail = `None`
- prefix/suffix/AIM/CodeID = `off`
- same barcode delay = `on`, `500-1000 ms`
- STM32 RX = `DMA + IDLE`
- STM32 dedup window = `1-2 s`
- boot mode = `Hybrid` (verify on boot, configure only on mismatch/uninitialized state)
- новый `PACKET_QR_GM810` публикуется через существующий `myQueueToMasterHandle`
- I2C окно для GM810 — отдельное fixed-length окно `32 B`
- `data`-поле окна = `28 B`, chunk-поля зарезервированы, но в v1 не используются

Это решение лучше всего соответствует текущей архитектуре проекта и не ломает существующий transport-подход STM32 -> ESP32.

## 13. Итог

GM810 хорошо вписывается в текущую архитектуру проекта, если рассматривать его как ещё один producer событий, аналогичный PN532, но работающий по UART.

Основная рекомендуемая схема такая:

- GM810 непрерывно ждёт код в фоне
- STM32 принимает UART-кадры
- STM32 выполняет локальную дедупликацию
- STM32 публикует событие в существующий I2C outbox для ESP32
- канал STM32 -> GM810 используется для boot-time verify/configuration и heartbeat

Аппаратурная база для первой реализации зафиксирована: `USART6` на `PA11/PA12`, с отказом от USB-функции в этом профиле.
